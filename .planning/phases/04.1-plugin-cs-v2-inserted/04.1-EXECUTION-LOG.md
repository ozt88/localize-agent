# Phase 4.1 Execution Log

## Bug 1: Choice text not translating (FIXED)

**Discovered:** During in-game verification
**Root cause:** D-14 assumption was wrong. Game sends choice text to `AddChoiceText` **pre-wrapped** with TMP tags:
```
<#FFFFFFFF><line-indent=-10%><link="N">N.   BODY</link></line-indent></color>
```
Plugin was trying to match the full tagged string against TranslationMap — always missed.

**Fix:** Added `ChoiceWrapperRegex` to extract body text from TMP wrapper, translate body only, re-wrap with original tags. Verified working in-game.

**Decision:** D-14 revised — game wraps BEFORE hook, not after.

## Bug 2: Glossary tag leak (NOT Plugin.cs bug — translation data issue)

**Discovered:** During in-game verification
**Symptoms:**
- `영광스러운 Quest</color>를 계속할 준비 완료.` — `</color>` visible
- `Norvik</color>,` — same pattern
- `crowns쯤은` — glossary color tag position broken

**Root cause:** Translation targets from LLM contain English glossary terms (Quest, Norvik, crowns) that the game's glossary highlighting system wraps with `<color>` tags. The opening `<color=...>` is processed by TMP but the closing `</color>` leaks as literal text, likely due to tag nesting conflicts with `<b>` tags already in the translation.

**Scope:** This is a translation data quality issue, not a Plugin.cs logic bug. Fix requires either:
1. LLM translator to translate glossary terms to Korean (so glossary system doesn't match)
2. Plugin to strip/re-apply glossary `<color>` tags (adds complexity)
3. Post-processing in pipeline to handle glossary term tagging

**Status:** Deferred — outside Phase 4.1 scope (Plugin.cs v2 rewrite)
