# Phase 04.2: 소스 정리 & 재export - Research

**Researched:** 2026-03-29
**Domain:** ink parser DC/FC strip, DB in-place update, export pipeline simplification
**Confidence:** HIGH

## Summary

Phase 04.2는 ink 파서에서 DC/FC 접두사를 strip하여 clean source를 생성하고, DB를 in-place UPDATE한 후 전체 패치 아티팩트를 재생성하는 작업이다. 동시에 export.go와 Plugin.cs에서 DC/FC 우회 로직을 제거하여 파이프라인을 단순화한다. LLM 재호출 없이 기존 번역을 재사용한다.

핵심 기술 도메인은 (1) 정규식 기반 텍스트 전처리, (2) SQL UPDATE with UNIQUE constraint 관리, (3) 기존 CLI 도구를 활용한 전량 재생성이다. 모든 코드가 이미 존재하는 도구 위에 구축되므로, 신규 라이브러리나 아키텍처 변경은 없다.

**Primary recommendation:** parser.go에 DC/FC strip 로직을 추가하고, 별도 Go 스크립트로 DB UPDATE + 재생성을 실행하며, export.go/Plugin.cs의 DC/FC 우회 코드를 제거한다.

### Pattern 1: 파서 DC/FC Strip

**What:** `walkFlatContent`의 `flushBlock` 함수 내에서 텍스트를 emit하기 전에 DC/FC 접두사를 strip하고, 능력치 타입을 tags[]에 추가한다.

**Where:** `parser.go` flushBlock() 내부가 최적 위치. 이유:
- 모든 블록이 이 함수를 통해 emit됨
- `tryExtractChoiceText`도 별도 블록 생성하지만, 선택지 텍스트가 `"s"` 배열에서 오므로 DC/FC 접두사가 포함될 수 있음 -- 두 곳 모두에 strip 필요

**Recommendation (Claude's Discretion):** 별도 함수 `stripDCFCPrefix(text string) (cleanText string, extraTags []string)`를 만들고, `flushBlock()`과 `tryExtractChoiceText` 둘 다에서 호출.

**Regex:** export.go에 이미 정의된 `dcfcPrefixRe = ^[A-Z]{2}\d+\s+\w+-` 를 그대로 사용. 단, 파서에서는 캡처 그룹이 필요:
- 그룹1: DC/FC 타입+번호 (예: "DC12", "FC8") -- 이미 tags[]에 있음
- 그룹2: 능력치 타입 (예: "str", "int") -- tags[]에 추가 필요

```go
// 제안 정규식 (캡처 그룹 포함)
var dcfcPrefixRe = regexp.MustCompile(`^([A-Z]{2}\d+)\s+(\w+)-`)
```

**Example:**
```go
func stripDCFCPrefix(text string) (string, []string) {
    m := dcfcPrefixRe.FindStringSubmatch(text)
    if m == nil {
        return text, nil
    }
    // m[1] = "DC12" or "FC8" (already in tags from ink # tag)
    // m[2] = "str", "int", etc. (ability type -- NEW, add to tags)
    body := text[len(m[0]):]
    return body, []string{m[2]}
}
```

### Pattern 2: DB In-Place UPDATE

**What:** 파서 재실행 -> parse_result.json 생성 -> 블록 ID 기반으로 DB의 source_raw/source_hash/ko_raw/ko_formatted을 UPDATE.

**Implementation approach (Claude's Discretion):** 별도 Go CLI 도구 `go-v2-source-cleanup/main.go`:

1. 파서 출력(parse_result.json)을 읽고 DC/FC가 strip된 블록만 필터링
2. DB에서 해당 블록 ID로 기존 레코드를 조회
3. 스냅샷 테이블에 기존 값 저장
4. UNIQUE 충돌 체크: 새 source_hash가 기존 다른 레코드의 source_hash와 겹치지 않는지 확인
5. source_raw, source_hash UPDATE
6. ko_raw, ko_formatted에서도 DC/FC 접두사 strip

**Snapshot table schema (Claude's Discretion):**
```sql
CREATE TABLE IF NOT EXISTS source_cleanup_snapshot (
    id TEXT PRIMARY KEY,
    old_source_raw TEXT NOT NULL,
    old_source_hash TEXT NOT NULL,
    old_ko_raw TEXT,
    old_ko_formatted TEXT,
    created_at TEXT NOT NULL
);
```

**UNIQUE 충돌 체크 (Claude's Discretion):**
```sql
-- 새 source_hash가 기존 레코드(다른 ID)와 겹치는지 체크
SELECT p.id, p.source_raw
FROM pipeline_items_v2 p
WHERE p.source_hash IN (... new hashes ...)
  AND p.id NOT IN (... IDs being updated ...);
```
충돌이 0건이면 안전하게 UPDATE 진행. 충돌이 있으면 중단하고 수동 조사.

### Anti-Patterns to Avoid
- **parser에서 DC/FC strip 누락:** `flushBlock()`만 수정하고 `tryExtractChoiceText`를 빠뜨리면 선택지 텍스트에서 DC/FC가 남음. 선택지가 DC/FC 체크의 주 대상이므로 반드시 두 곳 모두 적용.
- **ko_raw/ko_formatted UPDATE 누락:** source만 clean하고 target을 그대로 두면, target에 남은 DC/FC 접두사가 게임에 노출됨. D-07 명시적 요구.
- **UNIQUE 충돌 미체크:** clean source_hash가 기존 다른 레코드와 겹치면 UPDATE가 실패하거나 데이터 무결성 깨짐. 반드시 사전 체크.

## Common Pitfalls

### Pitfall 1: source_hash UNIQUE 위반

**What goes wrong:** DC/FC 접두사를 제거한 source가 이미 DB에 존재하는 다른 레코드의 source_raw와 동일해질 수 있다. 예: "DC12 str-Hello" strip -> "Hello", 그런데 "Hello"는 이미 별도 블록으로 존재.

**Why it happens:** 같은 텍스트가 DC/FC 체크 맥락과 일반 대사 맥락 양쪽에서 나타나는 경우.

**How to avoid:** UPDATE 전에 충돌 체크 쿼리 실행. 충돌 발견 시 해당 레코드만 스킵하고 수동 조사 (기존 레코드가 이미 번역 완료이면 DC/FC 레코드를 제거하는 것도 옵션).

**Warning signs:** UNIQUE constraint violation 에러 발생 시 즉시 중단.

### Pitfall 2: ko_raw에 DC/FC 접두사가 없을 수 있음

**What goes wrong:** LLM이 번역 시 DC/FC 접두사를 이미 제거했을 수 있다. "DC12 str-The Cleric?" 소스에 대해 ko_raw가 "성직자?"일 수 있음 (접두사 없음).

**Why it happens:** LLM 번역 동작이 결정론적이지 않음. 일부는 접두사를 포함하고 일부는 생략.

**How to avoid:** ko_raw/ko_formatted에 대해 dcfcPrefixRe를 적용하되, 매칭이 없으면 그대로 유지 (idempotent 연산).

### Pitfall 3: tryExtractChoiceText 경로 누락

**What goes wrong:** `walkFlatContent` > `flushBlock()`에만 strip을 추가하면, `tryExtractChoiceText`가 생성하는 블록에는 DC/FC가 남는다.

**Why it happens:** 선택지 텍스트는 `"s"` 배열에서 `extractTextFromArray`로 추출되어 별도 경로로 블록을 생성한다.

**How to avoid:** `tryExtractChoiceText`에서 블록 생성 전에도 `stripDCFCPrefix` 호출. 또는 공통 함수로 묶기.

### Pitfall 4: translations_loaded 회귀

**What goes wrong:** DC/FC body-only entry 제거 후 entries 수가 줄어, 일부 매칭이 실패할 수 있다.

**Why it happens:** 기존에 body-only entry로 매칭되던 항목이 이제 source가 clean하므로 직접 매칭 가능해야 하지만, 만약 파서 strip이 불완전하면 갭이 생김.

**How to avoid:** 재생성 후 translations.json entries 수를 기존 75,789와 비교. body-only entries (~579개) 제거분 빼고 유사해야 함. 인게임에서 `translations_loaded >= 75,789 - 579` 확인.

## Code Examples

### DC/FC Strip 함수 (parser.go)

```go
// Source: export.go dcfcPrefixRe 패턴을 캡처 그룹으로 확장
var dcfcPrefixRe = regexp.MustCompile(`^([A-Z]{2}\d+)\s+(\w+)-`)

// stripDCFCPrefix removes DC/FC stat-check prefix from text and returns
// the clean body text plus any new tags to add (ability type).
// The DC/FC tag itself (e.g., "DC12") is already in tags[] from ink # tag parsing.
func stripDCFCPrefix(text string) (string, []string) {
    m := dcfcPrefixRe.FindStringSubmatch(text)
    if m == nil {
        return text, nil
    }
    body := text[len(m[0]):]
    if body == "" {
        return text, nil // don't strip if body would be empty
    }
    // m[2] is ability type: "str", "int", "wis", etc.
    return body, []string{m[2]}
}
```

### flushBlock 수정 (parser.go)

```go
flushBlock := func() {
    text := textBuf.String()
    if text == "" {
        return

    // DC/FC strip: clean source and extract ability type tag

        Path:       path,
        Text:       cleanText,  // was: text
        SourceHash: SourceHash(cleanText),  // was: SourceHash(text)
        // ... rest unchanged
        Tags:       allTags,  // includes ability type
    }
    // ...
}
```
