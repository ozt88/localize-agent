# Phase 07.1: RAG+MCP 세계관 맥락 주입 - Research

**Researched:** 2026-04-10
**Domain:** RAG (Retrieval-Augmented Generation), wiki data indexing, translation pipeline context injection
**Confidence:** HIGH

## Summary

PageIndex(VectifyAI)는 vectorless RAG 시스템이지만, 인덱싱과 쿼리 모두 LLM API 호출이 필수다. 매 쿼리마다 LLM 호출이 발생하므로, 40,067건 x 배치당 1회 = 수천 회의 추가 LLM 호출이 필요하다. 이는 현재 파이프라인의 OpenCode 서버(custom session API, 비표준) 환경에서 비효율적이며, LiteLLM 호환성 문제도 있다.

**핵심 발견:** 프로젝트에 이미 위키 스크래퍼(`scrape_wiki.py`), 283페이지 위키 데이터(`wiki_pages/`), 254건 로어 텀뱅크(`esoteric_ebb_lore_termbank.json`)가 존재한다. GlossaryTerms.txt에는 800건의 용어 + 풍부한 설명이 있지만, 현재 `extractTermName()`이 설명부를 버리고 있다. v2pipeline은 현재 lore 통합이 전혀 없다 (v1 translation 패키지에만 존재).

**Primary recommendation:** PageIndex를 **오프라인 인덱스 빌드 전용**으로 사용하고, 런타임 쿼리는 Go 정적 매칭(기존 lore.go 패턴 확장)으로 수행. wiki + glossary 설명을 통합한 enriched termbank를 사전 생성하여, 배치별 JSON lookup으로 컨텍스트 주입. 또는 PageIndex 없이 기존 데이터만으로 구현 가능 (후자 권장).

## Critical Finding: PageIndex Architecture Analysis

### PageIndex가 "로컬 실행"이 아닌 이유

PageIndex(VectifyAI/PageIndex)는 다음 두 단계 모두에서 LLM 호출이 필수다: [VERIFIED: GitHub source code analysis]

1. **인덱싱 단계** (`build_index`/`page_index`): 문서를 hierarchical tree로 변환 — LLM이 TOC 추출, 섹션 구조화, 요약 생성
2. **쿼리 단계** (`query`): 트리 구조에서 LLM이 관련 노드를 reasoning으로 탐색 — **매 쿼리마다 LLM 호출**

```
인덱싱: PDF/MD → LLM(구조분석) → JSON tree → 저장 가능
쿼리:   질문 + tree → LLM(reasoning) → 관련 노드 ID → 텍스트 추출
```

### 프로젝트 환경과의 충돌

| 이슈 | 설명 |
|------|------|
| LLM API 호환성 | PageIndex는 LiteLLM 사용 (OpenAI /v1/chat/completions), OpenCode는 custom session API (비표준) |
| 쿼리당 LLM 비용 | ~4,000 배치 x 쿼리당 1회 = 4,000+ 추가 LLM 호출 (gpt-5.4 기준 상당한 비용/시간) |
| Python 의존성 | PageIndex는 Python 3.12+, LiteLLM, 추가 의존성 필요 |
| Ollama 호환성 가능 | LiteLLM은 Ollama를 지원하므로, Ollama 서버로 우회 가능하지만 추가 모델 로딩 필요 |

[VERIFIED: PyPI pageindex-open 0.1.1, GitHub VectifyAI/PageIndex source code]

### 권장 아키텍처: 2단계 하이브리드

D-06(PageIndex 확정)과 D-12(오프라인 우선) 결정을 존중하면서 실행 가능한 방안:

**Phase A: PageIndex로 오프라인 인덱스 빌드** (1회성)
- Ollama 로컬 모델로 PageIndex 인덱싱 실행 (인덱싱은 1회만 필요)
- 위키 283페이지 + glossary 800건을 tree structure로 변환
- 생성된 트리를 JSON으로 저장

**Phase B: 오프라인 사전 매칭** (1회성, Go 또는 Python 스크립트)
- 40,067건 전체 source_raw를 배치별로 그룹
- 각 배치 텍스트로 PageIndex 쿼리 실행 → top-3 결과 수집
- 결과를 `batch_id → [lore_hint_1, lore_hint_2, lore_hint_3]` JSON으로 사전 저장

**Phase C: Go 런타임 정적 로드** (파이프라인 실행 시)
- 사전 생성된 JSON을 Go worker가 로드
- 기존 glossary/voicecard와 동일한 패턴으로 ClusterTask에 주입

### 대안: PageIndex 없이 Enhanced Static Matching

기존 데이터만으로 더 단순하고 효과적인 구현이 가능하다:

| 데이터 소스 | 현재 상태 | 활용 방안 |
|-------------|-----------|-----------|
| `esoteric_ebb_lore_termbank.json` | 254 entries, term-match 준비됨 | lore.go 패턴 확장하여 v2pipeline에 통합 |
| `wiki_pages/*.json` | 283 pages, plain_text + wikitext 보유 | 섹션별 분리 후 extended termbank 생성 |
| GlossaryTerms.txt | 800 entries, 설명부 미활용 | 설명부 추출하여 lore 정보로 통합 |

이 방안은 PageIndex의 LLM 오버헤드 없이, 기존 lore.go의 word-boundary matching을 enriched termbank에 적용하는 것이다.

### Pattern 1: Enriched Termbank (통합 데이터)

**What:** wiki lore + glossary 설명부를 하나의 enriched termbank으로 통합
**When to use:** 데이터 전처리 단계에서 1회 실행

```go
// Source: 기존 lore.go + glossary loader.go 패턴 확장
type RAGEntry struct {
    Term        string   `json:"term"`
    Description string   `json:"description"`  // lore 또는 glossary 설명
    Category    string   `json:"category"`
    Source      string   `json:"source"`        // "wiki" | "glossary"
    Aliases     []string `json:"aliases,omitempty"`
    Section     string   `json:"section,omitempty"` // 위키 섹션 이름 (D-07)
}

type RAGHint struct {
    Term        string `json:"term"`
    Description string `json:"description"`
    Category    string `json:"category"`
}
```

**What:** trimContextForBudget에 RAG 우선순위 추가
**When to use:** 프롬프트가 토큰 예산 초과 시


### Anti-Patterns to Avoid

- **PageIndex per-query at runtime:** 매 배치마다 LLM 호출 발생 — 4,000+ 추가 호출, 파이프라인 지연
- **MCP/FastAPI 서버 상주:** 불필요한 복잡성 — 오프라인 사전 매칭이 충분
- **Wiki 데이터 재크롤링:** 이미 283페이지 보유, 게임 버전 고정이므로 재실행 불필요
- **Glossary 설명부를 별도 인덱스로:** wiki lore와 통합하여 단일 매칭 경로 유지

## Common Pitfalls

### Pitfall 1: PageIndex LLM 호환성

**What goes wrong:** PageIndex가 LiteLLM을 통해 OpenAI-compatible API를 기대하지만, OpenCode 서버는 custom session API 사용
**Why it happens:** OpenCode의 `/chat/send` 엔드포인트가 표준 `/v1/chat/completions`와 다름
**How to avoid:** PageIndex 인덱싱에는 Ollama 로컬 모델 사용 (11434 포트, LiteLLM Ollama provider), 런타임 매칭은 Go 정적으로 수행
**Warning signs:** `litellm.exceptions.APIConnectionError` 또는 JSON 파싱 에러

### Pitfall 2: 토큰 예산 초과

**What goes wrong:** RAG 컨텍스트 추가로 프롬프트가 4000 토큰 예산 초과
**Why it happens:** 기존 context (branch + continuity + voice + glossary) + RAG hints가 합산
**How to avoid:** D-18 우선순위에 따라 trimContextForBudget에 RAG 계층 추가, RAG hints를 간결하게 유지 (term: 1-2문장)
**Warning signs:** `estimateTokens(prompt) > contextBudgetTokens` 빈번 발생

### Pitfall 3: 설명부 품질 불균일

**What goes wrong:** wiki lore는 2문장(extract_first_sentence), glossary는 전체 설명 — 길이 편차가 큼
**Why it happens:** 데이터 소스마다 설명 길이/품질이 다름
**How to avoid:** enriched termbank 빌드 시 모든 설명을 2문장 이내로 정규화, 300자 상한 적용 (기존 build_lore_glossary.py의 패턴)
**Warning signs:** RAG hint 하나가 프롬프트의 대부분을 차지

### Pitfall 4: 매칭 노이즈

**What goes wrong:** 짧은 term (3글자 미만)이 false positive 발생 — "Ebb", "Ash" 등 일반 단어와 충돌
**Why it happens:** 게임 세계관 용어 중 일반 영단어와 겹치는 것이 있음
**How to avoid:** 기존 lore.go의 `len(term) < 3` 필터 유지, category 기반 우선순위 (character > location > lore > item)
**Warning signs:** 관련 없는 lore hint가 반복 등장

### Pitfall 5: GlossaryTerms.txt CSV 파싱

**What goes wrong:** 일부 행에 내장 줄바꿈이 있어 필드가 분리됨 (16번, 17번 행 등)
**Why it happens:** CSV의 Tags 컬럼에 줄바꿈 문자 포함
**How to avoid:** `csv.NewReader`에 `LazyQuotes = true`, `FieldsPerRecord = -1` 설정 (기존 loader.go 패턴)
**Warning signs:** 파싱된 행 수가 800보다 적음

```python
def extract_term_description(english: str) -> tuple[str, str]:
    """'Term - Description' 형식에서 term과 description을 분리"""
    for sep in [" - ", " – "]:
        idx = english.find(sep)
        if idx > 0:
            term = english[:idx].strip()
            desc = english[idx + len(sep):].strip()
            return term, desc
    return english.strip(), ""
```

## Open Questions

1. **PageIndex 인덱싱 품질 vs Ollama 모델 역량**
   - What we know: PageIndex는 기본 gpt-4o를 사용, Ollama 로컬 모델은 reasoning 능력이 낮을 수 있음
   - What's unclear: Ollama의 어떤 모델이 PageIndex tree-building에 충분한지
   - Recommendation: PageIndex 인덱싱은 선택적으로 수행. 기존 데이터(wiki_pages + lore_termbank + glossary)만으로 enriched termbank를 구축하면 PageIndex 없이도 D-06의 핵심 가치(세계관 맥락 주입) 달성 가능. **사용자 확인 필요: PageIndex LLM 비용을 감수할지, 아니면 기존 데이터 확장으로 대체할지**

2. **Glossary 설명부 중 게임 세계관과 무관한 항목**
   - What we know: GlossaryTerms.txt 800건 중 일부는 D&D 일반 규칙 설명 (예: "Ability Check", "Mage Hand")
   - What's unclear: 일반 규칙 설명도 번역 맥락으로 가치 있는지
   - Recommendation: 모두 포함 — 매칭되지 않으면 프롬프트에 나타나지 않으므로 해가 없음

3. **RAG가 score(평가) 프롬프트에 미치는 영향**
   - What we know: D-19에서 번역+평가 양쪽에 주입 결정
   - What's unclear: 평가 프롬프트의 토큰 예산과 구조가 다를 수 있음
   - Recommendation: 평가 프롬프트 구조를 확인하고 동일한 RAGHints 주입 패턴 적용
