# Phase 07.1: RAG+MCP 세계관 맥락 주입 — 위키 크롤링 + PageIndex 인덱싱 + 파이프라인 통합 - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

위키 데이터를 PageIndex(vectorless RAG)로 인덱싱하고, 번역 파이프라인에 세계관 맥락을 동적으로 주입. 기존 lore.go 정적 매칭을 RAG 기반 동적 검색으로 대체. MCP 레이어 없이 Go worker가 직접 호출하는 경량 아키텍처.

데이터 소스: 공식 위키 전량 + glossary 영문본 + 인게임 코덱스/저널 (확인 후)

</domain>

<decisions>
## Implementation Decisions

### 위키 데이터 수집
- **D-01:** 공식 위키 사이트에서 전량 크롤링 — 카테고리 선별 없이 모든 페이지 수집
- **D-02:** 크롤링 데이터를 페이지별 Markdown 파일로 저장 — PageIndex가 파일 단위로 인덱싱
- **D-03:** 모든 위키 페이지 포함 (메타 페이지 포함) — 인덱싱에서 관련도 낮은 페이지는 자연히 검색되지 않음

### 데이터 전처리
- **D-04:** 청정 본문만 추출 — 네비게이션, 사이드바, 푸터, 인포박스, 카테고리 링크 등 제거. 본문 + 제목만 유지
- **D-05:** 위키 페이지 중 제외할 유형 없이 모두 포함

### 인덱싱 + 검색 전략
- **D-06:** PageIndex 확정 (VectifyAI/PageIndex) — vectorless RAG, 로컬 실행
- **D-07:** 섹션 단위 인덱싱 — 페이지 내 헤딩별로 분리하여 더 세밀한 검색
- **D-08:** 배치 텍스트 직접 쿼리 — 배치 내 영어 소스 텍스트를 그대로 쿼리로 사용, PageIndex가 관련 로어 자동 매칭
- **D-09:** 검색 결과 Top 3 고정 — 기존 lore.go maxHints=3과 동일
- **D-10:** 기존 lore.go 정적 매칭을 RAG로 완전 대체 — 단일 경로

### RAG 아키텍처 (MCP 없이)
- **D-11:** MCP 레이어 없이 Go worker가 PageIndex를 직접 호출 — 현재 파이프라인이 자체 프롬프트 조립 구조이므로 MCP 불필요
- **D-12:** 오프라인 사전 매칭 우선, 불가 시 FastAPI 래퍼 폴백 (Claude 재량) — 게임 버전 1.1.3 고정이므로 배치 구성도 고정, 사전 매칭으로 충분
- **D-13:** PageIndex 서버/API 인터페이스는 리서치 단계에서 확인 후 최적 방식 결정 (Claude 재량)

### 추가 맥락 소스
- **D-14:** glossary 영문본 (GlossaryTerms.txt의 "Term - Description" 설명부) — RAG 인덱스에 통합, 통합 방식은 Claude 재량
- **D-15:** 인게임 코덱스/저널 — 게임 데이터에서 추출 가능하면 위키와 함께 인덱싱
- **D-16:** 기존 고품질 번역 참조는 이번 Phase에서 제외 — Phase 08 재번역 시 활용

### 파이프라인 통합
- **D-17:** RAG 결과를 [CONTEXT] 섹션에 주입 — 기존 glossary/lore와 동일한 위치, 4-tier 구조 유지
- **D-18:** 토큰 예산 우선순위: voice card > branch context > RAG ≈ glossary > continuity window — Phase 07 D-08 확장
- **D-19:** 번역(translateBatch) + 평가(score) 양쪽에 RAG 컨텍스트 주입 — 평가 LLM도 세계관 맥락 참조
- **D-20:** A/B 테스트 수행 — RAG 컨텍스트 주입 전/후 스코어 비교, Phase 07과 동일한 10개 배치 방식

### Claude's Discretion
- PageIndex API 인터페이스 확인 후 오프라인/서버 방식 최종 결정
- ClusterTask에 RAG 컨텍스트 필드 구조 (string vs struct slice)
- glossary 영문본의 RAG 인덱스 통합 방식 (설명부 추출 → 인덱싱 vs CSV 원본 vs Markdown 변환)
- 위키 크롤러 도구 선택 및 속도 제한 설정
- 인게임 코덱스/저널 데이터 존재 여부 확인 및 추출 방식

</decisions>

## Specific Ideas

- PageIndex가 오프라인 인덱스 저장/로드를 지원하면 사전 매칭 방식이 최적 — 배치별 위키 컨텍스트를 JSON으로 사전 생성하고 Go worker가 로드
- 기존 lore.go 대체 시 `matchedLoreEntries` → RAG 결과로 전환, `formatLoreHints` 포맷은 유사하게 유지
- GlossaryTerms.txt의 ` - ` 이후 설명 부분이 세계관 맥락으로 활용 가능 — 현재 extractTermName()이 버리는 데이터
