## MCP 서버 설계 → RAG 아키텍처 재설계

### MCP 필요성 재검토

User raised: "MCP가 꼭 필요한지 한번 더 확인"

분석 결과: 현재 파이프라인은 Go worker가 직접 프롬프트를 조립하는 구조이므로, LLM이 tool로 호출하는 MCP 패턴은 불필요. Go worker가 PageIndex를 직접 호출하면 MCP 레이어 없이 더 단순한 아키텍처 가능.

| Option | Description | Selected |
|--------|-------------|----------|
| Go worker 직접 호출 | translateBatch에서 PageIndex HTTP API 직접 호출 | ✓ |
| MCP 서버 경유 | MCP 서버를 두고 worker가 MCP 클라이언트로 호출 | |
| Python 서비스 (HTTP API) | PageIndex를 감싼 Flask/FastAPI 서버 | |

**User's choice:** Go worker 직접 호출

### 실행 모드 (오프라인 vs 실시간)

User asked: "오프라인 인덱싱에 대해 설명해줘", then "실시간 서버가 있으면 좋은점은?"

오프라인 장단점과 실시간 장단점을 상세 설명 후 결정.
